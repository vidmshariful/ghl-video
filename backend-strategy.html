Internal / StrategyFacts verified July 2026

One backend for the website: content, guardrails, and visibility.

The site is live-grade, but it is managed through a single 100KB TypeScript file that only a developer can touch. This document is the strategy for replacing that with a backend the whole team can use, without giving up the static architecture, the SEO work, or the HighLevel commerce stack.

01 / Why now

The content layer outgrew the file it lives in

Every word, price, SKU, FAQ, and section of ghlvideo.com lives in lib/site.ts: 2,651 lines, 100KB, roughly 50 exported blocks. It was the right choice for the build phase. One typed file meant prices could never drift between pages, and the launch QA proved that value repeatedly.

It is the wrong choice for operations. Three people who are not developers need to update videos, prices, reviews, and posts. Today every edit is a code change, a commit, and a deploy that only one person can safely make.

The evidence from launch week

Every serious defect the pre-launch review caught was a content-ops failure, not a code failure: a hero claiming a bundle floor the page itself contradicted, invented checkout slugs nobody had verified, a retired price surviving in a CTA string, year numbers hiding in card tags, a typo shipped inside SKU data. A backend with validation would have refused to publish all of them.

  • Speed: a price change or a new video should be a two-minute edit by anyone on the team, not a developer task.
  • Safety: the brand rules (pricing, vocabulary, no contractions, no year numbers) should be enforced by the system, not by memory.
  • Growth: the SEO roadmap requires publishing Knowledge Hub posts on a schedule. That is impossible if publishing means writing TypeScript.
  • Visibility: orders, bookings, and quote requests live in HighLevel. The team should see them next to the content that produced them.

03 / Hard non-goals

What this backend will never do

  • No payments. Checkout stays on order.ghlvideo.com funnels. The backend stores checkout URLs and verifies they respond; it never processes a dollar.
  • No CRM. Contacts, pipelines, and follow-up live in HighLevel. The dashboard reads them; it never becomes a second source of truth.
  • No booking engine. LeadConnector calendars own scheduling. The backend shows appointments, nothing more.
  • No runtime dependency for visitors. The public site stays statically generated. If the backend is down, the website does not care.
  • No second design system. The admin edits content that renders through the existing components. Editors choose content, never layout.
The principle

HighLevel is the system of record for money and people. The backend is the system of record for content. The website is a fast, static projection of both. Every integration is read-only in exactly one direction.

06 / Recommendation

Phase zero costs nothing. Then Payload as a sidecar.

Phase zero, this week, no new infrastructure: keep content in git and make it safe. Split lib/site.ts into domain files under /content, move the guardrails validators (section 08) into CI so a rule violation fails the build, and stand up the daily checkout-link checker on GitHub Actions. This delivers most of the safety value of a backend for zero dollars and zero migration risk, and nothing built in phase zero is thrown away later: the validators and checks become the CMS publish hooks.

Then the real backend: Option A, Payload 3 as a sidecar admin. The reasoning, in order of weight:

  • The site stays untouched. The static export, the SEO layer, the five #1 rankings, the Cloudflare portability: none of it is renegotiated. The backend is additive, not surgical.
  • Free means free. Payload is MIT (and actively developed after the Figma acquisition; v3.86 shipped this month). Neon's free Postgres auto-wakes on connection, which suits a database that is only touched when someone edits content or a build runs. Software cost stays zero at any team size.
  • The catalog is relational. SKUs reference packs, packs reference bundles, bundles reference prices. Postgres relations plus Payload's admin UI handle this the way a document store only approximates.
  • The types survive. Payload schemas are code-first TypeScript. The collections in section 07 are close to a transcription of the types that already exist, and the seed import is a script over the current file.

Runner-up: Sanity, if after seeing both admins you prefer zero infrastructure over zero vendor. Everything else in this document works identically with either choice; only section 07's implementation notes change.

08 / Guardrails engine

The rulebook becomes software

Every rule below was violated at least once during the build phase and caught by manual review. In the backend they run as publish-time validators: a save that breaks one is rejected with the reason, before it can reach the site.

ValidatorRule enforced
CharactersNo em dashes, en dashes, ellipsis characters, or middle dots in any visible field.
ContractionsRejected everywhere except fields flagged as verbatim quotes.
YearsNo year numbers in any customer-facing field.
Proof figuresClient count must be the literal 800+; rating the literal 5.0. No other figures accepted.
Pricing integrityPrices live only in the plans and offers collections; copy fields reject dollar amounts that do not match a referenced price. Meta descriptions and titles reject dollar amounts entirely.
CTA vocabularyCTA fields accept only: Book a Call, Order Now, Start editing, Request a Quote, See the plans, See the videos.
Checkout URLsMust match order.ghlvideo.com, and must have passed the most recent daily link check before publish.
Media weightPreview uploads over the size ceiling are rejected; a video without a poster cannot publish.
SEO shapePosts require focus keyword, meta within length, direct-answer intro, and one internal link before publish.
AffiliationAny field mentioning HighLevel endorsement or partnership requires founder-role approval to publish.
09 / Publish pipeline

From draft to live in one safe path

  1. Draft. Editor changes content in the admin. Versions retained; nothing touches the site.
  2. Validate. Guardrails run on save. Violations block with a plain-language reason.
  3. Preview. A preview build renders the draft on a private URL for sign-off.
  4. Publish. The admin fires the Vercel deploy hook (verified: hooks are on all plans, rate-limited at 60 per hour, roughly three cents of build cost per publish).
  5. Acceptance. The existing QA suite runs against the fresh build: rendered pages, dead links, copy-rule scan, schema validation. A red run rolls back to the previous deployment.
  6. Log. Who published what, when, and what the validators said. The audit trail the file never had.
Reuse, not rebuild

Step five is the playwright and python QA harness already written during launch week, promoted from a developer habit to a pipeline gate.

10 / Ops dashboard

HighLevel visibility, read-only, verified feasible

The API research confirmed every panel below is buildable today with a Private Integration Token: no marketplace app, no OAuth flow, no commerce rebuild.

PanelSource (API 2.0)Notes

Quote and form leadsGET /forms/submissionsPlus surveys if used.
Checkout-link healthDaily GitHub Actions checkHEAD-checks all 28 funnel URLs; failures alert before a buyer finds them.
  • Auth: a sub-account Private Integration Token with read-only scopes, rotated quarterly. Verified limits: 100 requests per 10 seconds, 200,000 per day. Polling every few minutes uses under 1 percent of that.
  • Push later, if wanted: native webhook topics require a marketplace app, but workflow triggers (Order Submitted, Payment Received, Customer Booked Appointment, Form Submitted) can fire standard outbound webhooks at the dashboard. Start with polling; add push only if minutes matter.
  • Security line: the token lives server-side in the admin app only. It can never ship in the static site bundle, because a static export ships everything to the browser.

12 / Rollout plan

Five phases, each shippable alone

  1. Phase 0: safety in git (days). Split site.ts into /content modules, validators into CI, link checker on Actions, Vercel plan verified. Acceptance: a rule violation fails the build; a dead checkout URL alerts within a day.
  2. Phase 1: the sidecar admin (week one). Payload app + Neon, collections from section 07, seed import from the current file. Acceptance: every SKU, page block, and price visible and editable in the admin; the site still builds from git, untouched.
  3. Phase 2: the switchover (week two). The site's build fetches from Payload instead of the local file, with the file kept as a frozen fallback for one release. Publish wired to the deploy hook and the acceptance suite. Content freeze during the switch; the old file is deleted only after two clean releases. Acceptance: an editor changes a price in the admin and it is live, validated, within minutes, with zero developer involvement.
  4. Phase 3: Knowledge Hub (week three). Posts collection live with the SEO-shape validator; the first three posts from the SEO guide published through it. Acceptance: a post goes from draft to indexed without a developer.
  5. Phase 4: ops dashboard (week four). PIT created, orders, bookings, and leads panels live next to the content. Acceptance: the team answers "did we sell anything today" without opening HighLevel.

14 / Risks

What can go wrong, and the mitigation

  • Dual-source drift during migration. Mitigation: content freeze during phase 2, the file kept as read-only fallback, deletion only after two clean releases.
  • The admin becomes a second website project. Mitigation: the non-goals in section 03 are scope law. The admin gets no design system, no public pages, no ambitions.
  • Free-tier surprises. Mitigation: Neon auto-wakes (verified) so a quiet database is safe; the only hard dependency at visitor time is static hosting, which has no tier risk.
  • Token leakage. Mitigation: PIT is server-side only, read-only scopes, quarterly rotation. The static site never sees it.
  • Single-maintainer bus factor moves rather than disappears. The admin still needs a developer for schema changes. Mitigation: schemas change rarely once modeled; day-to-day operation needs no developer at all, which is the point.
15 / Decisions needed

Five calls only the founder can make

  1. Confirm the Vercel plan. Hobby is non-compliant for this site. Pro, or move static hosting to Cloudflare.
  2. Payload sidecar or Sanity. The recommendation is Payload; both admins can be demoed with the real catalog in a day each before committing.
  3. Approval mode at start. Editor-publishes-within-guardrails, or editor-drafts-founder-approves for the first months.
  4. Ops dashboard scope. Orders and bookings only, or leads too.
  5. Phase 0 start. It costs nothing, breaks nothing, and can begin immediately.
